# Universe Selection

## The problem: which symbols to trade?

A trading system needs a **universe** — the set of symbols it will analyze and potentially trade. Two extremes are both problematic:

- **Too narrow** (e.g., just 10 symbols): misses opportunities, over-concentrates risk
- **Too broad** (e.g., all 10,000 US-listed equities): most are illiquid, too small, or too noisy to model reliably; data costs and compute time become prohibitive

The goal is a focused universe of the most tradable, highest-quality candidates — typically 50–100 symbols — that changes slowly over time as the market evolves.

---

## Static watchlist vs dynamic universe

### Static watchlist (simple, default)
A hand-picked list of symbols in `config.data.watchlist` (default: 10 large-caps). This never changes unless you edit it manually. Good for getting started, bad for capturing the full opportunity set.

### Dynamic universe (automated)
When `config.universe.enabled = True`, the pipeline replaces the static watchlist with a dynamically selected set of candidates generated by the three-stage funnel described below. The universe is refreshed weekly (full refresh) and rescored daily.

---

## The three-stage funnel

```
All US-listed equities (via Alpaca)    ~10,000 symbols
              │
              │ Stage 1: Eligibility filter
              ▼
        Tradable, active US equities   ~8,000 symbols
              │
              │ Stage 2: Liquidity & size filter
              ▼
        Liquid, mid-to-large cap        ~300 symbols
              │
              │ Stage 3: Momentum + liquidity rank-percentile
              ▼
        Top 50 candidates               50 symbols
              │
              │ + Permanent fixtures (always included)
              ▼
        Active universe                 ~68 symbols
```

---

## Stage 1: Eligibility (Alpaca API)

`UniverseSelector._stage1_fetch()` calls Alpaca's `get_all_assets()` endpoint to retrieve all listed US equities. Filters applied:

```python
assets = [a for a in all_assets if
    a.status == "active"         and  # currently trading
    a.tradable == True           and  # Alpaca can trade it
    a.asset_class == "us_equity" and  # equities only (no crypto)
    a.exchange in allowed_exchanges   # NYSE, NASDAQ, ARCA, BATS, AMEX
]
```

The exchange filter (`allowed_exchanges`) excludes OTC markets, which eliminates most foreign ADRs and pink sheet stocks. OTC equities are often illiquid and subject to different regulatory requirements.

**Requires:** `ALPACA_API_KEY` and `ALPACA_SECRET_KEY` environment variables. Without keys, Stage 1 raises `UniverseError` and only permanent fixtures are included.

---

## Stage 2: Liquidity and market cap filter

Stage 2 eliminates symbols that are too small or illiquid to trade reliably. Two filters:

### Market cap floor

```python
market_cap = fundamentals.get("market_cap")
if market_cap < min_market_cap:   # default: $1 billion
    drop(symbol)
```

Stocks below $1B market cap are typically:
- More volatile (fewer institutional investors to stabilize prices)
- Less liquid (larger bid-ask spreads, harder to fill at expected price)
- Less covered by analysts (less news for FinBERT)
- More susceptible to manipulation

### Average dollar volume floor

```python
# Average (close × volume) over the last 20 bars
avg_dollar_volume = (bars.Close × bars.Volume).mean()
if avg_dollar_volume < min_avg_dollar_volume:   # default: $5 million/day
    drop(symbol)
```

Dollar volume (not share volume) is the right metric because it's comparable across different price levels. A stock trading 1 million shares/day at $1 has the same dollar volume as one trading 100 shares at $10,000 — but very different liquidity profiles for a typical order size.

$5M/day ensures the system can enter and exit positions without significant market impact.

---

## Stage 3: Momentum + liquidity rank-percentile

Stage 3 ranks the surviving ~300 candidates and selects the top `stage3_max` (default 50) on a transparent two-factor blend:

```python
score = 0.5 × pct_rank(20-day return)
      + 0.5 × pct_rank(avg_dollar_volume)
```

Both inputs use **rank-percentile** rather than the raw value — that means:
- A single mega-cap with $10B/day ADV doesn't dominate the score scale; it just gets the top liquidity rank.
- Scaling either input by a constant (or any monotonic transform) doesn't change the ordering.
- The score is always in `[0, 1]` regardless of input units.

Symbols missing OHLCV bars after the pre-stage backfill (fewer than 21 cached bars) get the worst momentum rank automatically.

### Why not an ML model?

The previous Stage 3 ranker loaded one symbol's XGBoost checkpoint (whichever was first alphabetically in `models/cache/`) and applied it as a universal classifier to every other Stage 2 survivor. That's not a meaningful question — the per-symbol model only learned its own price/indicator history, not a general "is this stock attractive" function. The 2026-05-10 weekly run dropped INTC, BA, AZN, VALE, and UAL — five of the universe's top WF-Sharpe symbols — because the AAOI model rated them poorly.

A proper cross-sectional ranker (a model trained explicitly on `(symbol, current_indicators) → forward return` pairs across the whole universe) would be a multi-day project; for the role Stage 3 actually plays — pre-filtering 300 candidates down to 50 worth training deep models on — momentum + liquidity is enough. The per-symbol ensemble in walk-forward is what's supposed to predict returns.

## Permanent fixtures

Certain symbols are **always included** in the universe regardless of funnel results:

```yaml
permanent_fixtures:
  - SPY    # S&P 500 ETF
  - QQQ    # Nasdaq ETF
  - IWM    # Russell 2000 ETF
  - DIA    # Dow Jones ETF
  - XLF, XLE, XLK, XLV, XLI, XLP, XLY, XLU, XLB, XLRE  # sector ETFs
  - TLT    # 20+ year Treasury bonds
  - GLD    # Gold
  - SLV    # Silver
  - USO    # Oil
```

These fixtures serve two purposes:
1. **Diversification**: sector ETFs and macro assets ensure the portfolio isn't purely equity-concentrated
2. **Regression testing**: SPY and QQQ are always present, making it easy to compare signals against market benchmarks

Fixtures bypass all three funnel stages and are never removed by the scoring process.

## Scheduling

| Run | When | What happens |
|-----|------|-------------|
| Full refresh | Sunday 01:00 | All three stages; new candidates replace old universe |
| Daily rescore | Mon–Sat 05:30 | Stage 3 only; re-ranks existing candidates by current momentum + ADV; doesn't add new symbols |

The daily rescore is fast (seconds) because it skips the expensive Alpaca API call (Stage 1) and liquidity filter (Stage 2). It just re-ranks the already-approved candidate pool with the latest 20-day momentum and dollar-volume figures.

### Why not refresh daily?

Stage 1 (Alpaca) is rate-limited and could return different symbols day to day based on temporary trading halts or data issues. A weekly full refresh strikes the right balance: the universe stays relatively stable (fewer handoffs in active positions) while still adapting to the market monthly.

## Database tables

### `universe_assets`

One row per candidate symbol:

| Column | Description |
|--------|-------------|
| `symbol` | Ticker (primary key) |
| `active` | Currently in the universe? |
| `is_fixture` | Permanent fixture (bypasses funnel)? |
| `stage` | Which stage it passed through (1, 2, or 3) |
| `market_cap` | Market capitalization from fundamentals |
| `avg_dollar_volume` | 20-bar average dollar volume |
| `stage3_score` | Stage 3 rank-percentile blend of 20d return + ADV, in [0, 1] |
| `added_at` | When it entered the universe |
| `last_scored_at` | When it was last rescored |
| `removed_at` | When it was removed (if not active) |

### `universe_run_log`

One row per stage per run — tracks how long each stage took and how many symbols passed through. Visible on Page 7 of the dashboard.

---

## Universe page (Page 7)

The dashboard's Universe page shows:

**Funnel chart**: how many symbols passed each stage (Stage 1 → Stage 2 → Stage 3 → active). A wide funnel with a narrow bottom means the filters are working as intended.

**Active candidates table**: current universe with scores, market cap, and dollar volume. Sortable by any column.

**Size history**: how many symbols were in the universe over time. A sudden drop might indicate a Alpaca API issue or an unusually strict rescore.

**Recently removed**: symbols that were in the universe but dropped out. Useful for spotting symbols that degraded in quality or liquidity.

**Manual controls**: sidebar buttons to trigger a full refresh or rescore immediately, without waiting for the scheduled task.
