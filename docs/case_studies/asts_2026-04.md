## 4. Skill vs luck retrospective

### 4.1 Did the system's risk machinery behave as designed?

**Mostly yes — with one structural failure.**

- ✅ Position sizer correctly returned 565 shares — comfortable but not excessive (~4 % of equity).
- ✅ Bracket stop at $56.96 (2.0 × ATR below entry) survived the -$4.55 drawdown on 5/5 (low $66.15 was $9.20 above stop). Stop never triggered.
- ✅ Bracket TP at $91.32 (3.0 × ATR above entry) never fired either — today's high $89.65 was $1.67 below TP. The signal-flip path closed the position before TP. (Compare to AXTI which gapped through TP at $97.44 open → $101.19 fill before the manager ran. ASTS's gap-up today was $87.67 open — well below TP.)
- ✅ Trail conversion correctly fired in Phase 3.5 once the prior-bar close exceeded activation. The activation arithmetic was right.
- ✅ Long-only SELL gate (`allow_short_selling=False`) correctly routed to `_close_long_position` rather than opening a short.
- ❌ **`_cancel_bracket_children` does not include TRAIL in its filter list.** This is a real bug, surfaced today for the first time because today is the first observed case of `(trail-converted) AND (CLOSED_LONG via signal-flip)` in the same run. SNOW converted on 5/7 but never flipped; AON converted on 5/19 but never flipped; AXTI gapped through TP. ASTS is the new code path. (The orphan TRAIL id=173 was manually cancelled at 10:37 ET via `scripts/open_orders.py --cancel --id 173` — confirmed by the cancel response `cancel sent for id=173` and `Remaining open: 24`. Code-level fix remains pending and is the headline CLAUDE.md "Outstanding bugs" proposal from today's review.)
- ❓ The trail activation was driven by Phase 2's mid-morning partial-bar refresh — `ohlcv_bars` recorded $87.28 as today's "close" at 10:06 ET, before the regular session even opens. This is the **same partial-bar-becomes-permanent issue** that motivated the EOD refresh script (CLAUDE.md "Stale partial-day bars" — code complete 2026-05-19). In this case the partial bar *helped* (it surfaced the intraday print to the trail manager) but the architectural inconsistency stands: the trail manager is implicitly evaluating against an intraday quote when the user expects EOD-only logic.

### 4.3 Counterfactuals — would a different parameter setting have changed the outcome?

Three knobs to model:

**A. Lower `trailing_stop_activation_atr` from 2.0 → 1.5.** Activation would have been $70.71 + 1.5 × 6.47 = $80.42 on 5/11. The 5/11 close ($80.22) would have just barely missed, but the 5/14 close ($75.48) clearly below. The 5/15 close ($78.61) → 5/18 morning evaluation against close=$78.61 → activation 1.5×6.56 = $80.55 → still SKIPPED. 5/19 morning against 5/18 close $82.25 → activation 1.5×6.56 = $80.09 → CONVERTED. Trail distance 1.5×6.56=$9.84. 5/19 low $80.40 vs trigger $82.25 - $9.84 = $72.41 → safe. 5/20 high $89.65 → ratchet trail trigger to $79.81. Then if SELL still fires today, MKT close at $87.28 = +23.4 % (same outcome, just orphans a different trail). If SELL doesn't fire today, trail keeps ratcheting — possibly exits slightly later at a price within ±$3 of today. **Net: roughly equivalent for this trade.** Cost: lower activation means trail engages on weaker moves that may retrace — would have hurt AXTI (premature activation pre-gap-up).

**B. Raise `atr_take_profit_multiplier` from 3.0 → 5.0.** TP would have been at $70.71 + 5.0 × 6.87 = $105.05 instead of $91.32. Today's high $89.65 was below either TP — no change for ASTS. But would have helped AXTI (TP would have been at $107.31, above the $101.19 actual fill, so trail would have engaged instead). **Net: helpful for AXTI's archetype, irrelevant for ASTS.**

**C. Intraday trailing-stop evaluation (e.g. additional 12:00 ET and 15:30 ET runs of Phase 3.5 only).** This is the most consequential knob. Today's TRAIL conversion would have happened the same way (the morning run already caught it via partial-bar refresh). But on 5/18, intraday high $89.96 (well above activation $83.85) would have triggered CONVERTED at ~$89.96 in the 12:00 ET slot. Trail distance $13.35 → trigger $76.61. 5/18 close $82.25 — well above trigger. 5/19 low $80.40 — still above. 5/20 high $89.65 — would ratchet trail trigger to $76.30. Then today's SELL signal still fires in Phase 4 → MKT close at $87.28 = same outcome. **For ASTS specifically, intraday evaluation is a wash — the morning partial-bar refresh already captures the spirit of intraday for the trail.** But for the AXTI archetype (gap-through-TP at the open), intraday is irrelevant; only "TP becomes trailing once first-target is hit" would have helped AXTI.

### 4.4 The one-sentence read

**Skill on the risk machinery (bracket stop survived a -$4.55 drawdown and a -11 % earnings gap-down), structural luck on the exit timing (Phase 2's partial-bar refresh surfaced today's intraday print as the trail's activation input, while Phase 3's SELL signal happened to fire on the same morning at the same price), and one real bug — the Phase 4 close path doesn't cancel TRAIL orders, so today left an orphan at IBKR that needs manual cancellation.**

### New finding — Phase 4 long-only close does not cancel TRAIL orders

Today is the first observed instance of `(Phase 3.5 TRAIL placed) AND (Phase 4 CLOSED_LONG fired)` in the same run. The `_cancel_bracket_children` filter in `risk/order_manager.py` lines 384-389 reads:

```python
targets = [
    o for o in open_orders
    if o.get("symbol") == symbol
    and o.get("action") == "SELL"
    and o.get("order_type") in ("LMT", "STP", "STP LMT")
]
```

TRAIL is missing. The Phase 3.6 hold-timeout codepath (which uses a broader filter `("LMT", "STP", "STP LMT", "TRAIL")` per CLAUDE.md notes) already gets this right; Phase 4 doesn't. Proposed CLAUDE.md "Outstanding bugs" entry:

```
- **Phase 4 long-only close doesn't cancel TRAIL orders, leaving orphans after same-run trail conversion + signal flip** (`risk/order_manager.py:384-389`): when a position has a trailing stop converted in Phase 3.5 AND the same run generates a SELL signal that routes to `_close_long_position` in Phase 4, the bracket-child cancel filter only includes `("LMT", "STP", "STP LMT")` — TRAIL is missing. The MKT close flattens the position but the TRAIL order stays live at IBKR; when price subsequently drops by the trail distance, IBKR fires SELL against a zero-share position and may open an unintended SHORT depending on account permissions. Observed 2026-05-20 on ASTS: Phase 3.5 placed TRAIL id=173 at 10:08:41, Phase 4 placed MKT id=179 at 10:08:44 to close, TRAIL id=173 was not cancelled — verified by absence of any `Cancel request sent for order_id=173` in `logs/daily/daily_run_20260520.log` and by the cancel-filter source. The Phase 3.6 hold-timeout path uses a broader filter that already includes TRAIL, so the fix pattern exists in-repo. **Fix:** add `"TRAIL"` to the `order_type in (...)` tuple in `_cancel_bracket_children`. Add a regression test in `tests/test_signal_runner.py` (or `test_risk.py`) that simulates a held position with an active TRAIL order, fires a SELL signal in long-only mode, and asserts the cancel list includes the TRAIL. **Manual remediation needed today:** cancel id=173 via `scripts/open_orders.py --cancel --id 173`.
```

### New finding — Trail manager implicitly uses intraday quotes via Phase 2's partial-bar refresh

The trail manager was *intended* to be a daily-bar evaluator — but in practice, Phase 2 fetches yfinance at ~10:06 ET (post-open, intraday) and writes the current quote as today's "1d close" in `ohlcv_bars`. The trail manager (Phase 3.5) then reads that as the latest close — which is an intraday print, not an EOD close. Today this *helped* (caught the rally that prior-day-close evaluation would have missed). But the same mechanism would also cause spurious activation if the morning print spikes briefly above activation and then retraces. Worth flagging because it muddles the "daily-bar vs intraday" framing in the trailing-stop crowded-out enhancement.

```
- **Trail manager implicitly uses intraday quotes when Phase 2 refresh runs mid-morning** (`risk/trailing_stop.py`, `scripts/signal_runner.py:_phase2_data_refresh`): Phase 2 calls yfinance at ~10:06 ET, after the regular session has opened, and writes the live intraday quote as today's "1d close" in `ohlcv_bars`. Phase 3.5 then reads `get_bars(symbol, "1d", limit=1)` and uses that intraday value as the price input to the activation check. The trail manager's docstring (and CLAUDE.md "Trailing stops run as Phase 3.5") imply EOD-close semantics, but in practice the input is an intraday print. **Observable consequence (2026-05-20 ASTS):** trail activated at 10:08:41 against partial-bar close $87.28, even though Mon's session was still mid-morning. Had Phase 2 not run today, the manager would have used Tuesday 5/19's $80.57 close → activation $83.69 → SKIPPED → trail never converts. The trade still exits via SELL signal in Phase 4 either way for this case, but the principle is that ~50 % of recent trail evaluations have been driven by Phase 2's partial-bar refresh rather than the prior-day close. **Fix considerations:** (a) document the current behaviour explicitly in CLAUDE.md and `trailing_stop.py` docstring (cheapest); (b) refactor the trail manager to read the *prior* daily bar via `get_bars(symbol, "1d", limit=2)[0]` so EOD-close semantics are enforced (more conservative, would have left ASTS unactivated until the EOD refresh after-market — which is now landed via `run_eod.bat`); (c) move trail evaluation explicitly to an intraday cadence (the "Intraday lightweight runner" enhancement) so the input timing is by design rather than incidental. Option (a) is a 15-minute change; (c) is the principled fix.
```
