## 2026-05-01

### Long-only close didn't cancel bracket children, leaving an orphaned STP that opened an unintended short

*(code complete 2026-05-01 â†’ verified retroactively 2026-06-04 via the SLV 2026-04-29 forensic reconstruction)* (`risk/order_manager.py:_close_long_position` / `_cancel_bracket_children`): before this fix, the long-only SELL path (`_close_long_position`) market-sold to flatten the position but did **not** cancel the entry bracket's still-live GTC children (the STP stop-loss and LMT take-profit legs). Those legs survived at IBKR against a now-zero-share position; a subsequent price move that triggered the orphaned STP would SELL into flat and **open an unintended short** â€” directly violating `allow_short_selling=False`. This is the *root* orphan-cancellation gap; the later 2026-05-21 entry ("Phase 4 long-only close doesn't cancel TRAIL orders") is a refinement of the *same* fix that extended the cancel filter to TRAIL legs.

**Canonical example â€” SLV, reconstructed 2026-06-04 from `fill_log` + `order_decisions` + `logs/daily/daily_run_20260427.log`:** (1) **2026-04-22** BUY 566 bracket entry, APPROVED, STP stop @ ~$65.08. (2) **2026-04-27** SELL signal flip â†’ `CLOSED_LONG`; the daily log shows the close went straight to `Placing MARKET order: SELL SLV x566` â†’ `Market order result: [Filled]` â†’ `CLOSED_LONG` with **no `Cancel request sent` line at all** â€” because `_cancel_bracket_children` did not yet exist on that date (the close pre-dated this very commit by 4 days). The STP leg was left live. (3) **2026-04-29** SLV fell to $64.96, tripping the orphaned STP â€” `fill_log` records the fill as `order_type='STP'`, 566 shares SELL â€” which opened a 566-share short. (4) **2026-04-30** BUY 566 MKT covered the short. The resulting `trade_log` round trip (`id=2002`) was a genuine short that the long-only reconciliation aggregator then mislabeled with reversed entry/exit timestamps (see the separate 2026-06-04 data-correction note); the short itself was this bug.

**Status:** `_cancel_bracket_children` introduced in commit `06b5361` (2026-05-01, bundled in the "Committing several days of code updates" commit â€” which is why this fix had no dedicated CHANGELOG entry until now). `_close_long_position` calls it before submitting the market close, cancelling open SELL bracket legs of type `("LMT", "STP", "STP LMT")` for the symbol. Commit `73aceb5` (2026-05-21) widened that filter to `("LMT", "STP", "STP LMT", "TRAIL")` â€” see the 2026-05-21 entry for the TRAIL-specific test (`test_cancel_bracket_children_cancels_trail_orders`) and the ASTS production incident.

**Verified (retroactively, 2026-06-04):** no SLV-class orphan short has recurred since the fix. Every SLV long-close after 2026-05-01 cancels its bracket children, and the only orphan-short round trip in `trade_log` (`id=2002`) is the pre-fix 2026-04-29 event whose entry pre-dated the cancellation feature. The detection query `SELECT * FROM trade_log WHERE exit_ts < entry_ts` (which surfaced the SLV row) returns zero rows after the data correction. Forward regression protection lives in the 2026-05-21 `test_cancel_bracket_children_cancels_trail_orders` test, which pins the cancel-filter membership.

**Correction (2026-06-05) â€” the 2026-06-04 "Verified" claim above was premature; the exit-before-entry corruption *regenerated* on the 2026-06-05 daily run.** The `_close_long_position` cancellation fix (the actual subject of this entry) is sound and unaffected â€” no *new* orphan short has been created since 2026-05-01. But the 2026-06-04 work also did a one-off **data correction** to `id=2002` (re-labelling it as a proper short with `entry_ts < exit_ts`), and that correction did **not** fix the *reconciliation aggregation logic* that produced the corrupt row in the first place. Worse, by changing `id=2002`'s `exit_exec_id` to the 4/30 MKT-buy leg, it freed the 4/29 STP-sell leg to be re-claimed: on 2026-06-05 `execution/reconciliation.py:_aggregate` re-paired the **same two April orphan-short fills** in the opposite direction (a "long" with entry 4/30 â†’ exit 4/29) and wrote `trade_log` id=2022 â€” `exit_ts < entry_ts` again, double-counting the âˆ’$974.39 loss against id=2002. The `uq_trade_live_exit` partial unique index keys only on `exit_exec_id`, so the reverse pairing (different exit leg) dodged dedup. **The generating logic was fixed 2026-06-05** in `execution/reconciliation.py:_write_round_trip`: (a) a hard `exit_ts <= entry_ts` guard rejects + WARNs any inverted pairing (`ReconcileResult.n_skipped_inverted`), and (b) a both-legs dedup via the new `data.database.live_trade_uses_exec_ids` â€” a fill already consumed as **either** an entry or exit leg of a live row can no longer be re-paired. `id=2022` was deleted; `WHERE exit_ts < entry_ts` returns zero rows again. Regression protection: `tests/test_reconciliation.py::TestInvertedRoundTripGuard` (2 tests) pins both the no-write and the stable-on-rerun behaviour. **Lesson: a data correction is not a fix** â€” this orphan-short reconciliation work stays out of "verified" until the *logic* fix has survived several daily runs without regenerating the row (tracked in `docs/reviews/followups.md`, 2026-06-05).
