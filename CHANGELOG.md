### Phase 4 stacks a second bracket on a symbol with an unfilled GTC entry already working

*(code complete 2026-06-15 â†’ verified 2026-06-25)* (`scripts/signal_runner.py:_fetch_pending_entry_symbols` + `signal_runner_log.skipped_pending_orders`; commit `8465563`): bracket entries are submitted **GTC**, so a BUY LMT the symbol gaps past at the open sits unfilled for days. `PortfolioGuard`'s duplicate check only sees *filled* positions (`_fetch_positions` â†’ `quantity != 0`), so without a separate guard the next daily run that regenerates the same signal would submit a **second** bracket on top of the still-working unfilled entry â€” two live entry brackets on one symbol. **Motivated by HPE + LRCX 2026-06-15:** two Friday BUY LMTs gapped past Monday's open and sat unfilled GTC; they had to be cancelled manually before the run to avoid a double-stack. This is the live-order analogue of the within-session `EQUIVALENT_PAIRS` dedup, but against the *broker's* open-order book rather than within-run state.

**Status:** implemented (commit `8465563`). Phase 4 calls `_fetch_pending_entry_symbols(ibkr, loop, positions)` (one `get_open_orders()` call) to build the set of symbols with a working open order but **not** currently held â€” held symbols are excluded because their open orders are protective TP/STP/TRAIL legs (covered by the duplicate guard + long-only close path) â€” and skips any actionable signal (or its `EQUIVALENT_PAIRS` partner) in that set. **Best-effort:** returns an empty set when IBKR is unavailable, so it never hard-blocks submission. The per-run `skipped_pending_orders` count is printed in Phase 5 and persisted to `signal_runner_log` (idempotent `_migrate()` ALTER). Test coverage: 10 unit tests in `tests/test_signal_runner.py`. Documented as the "Phase 4 dedups against unfilled entry orders, not just filled positions" architectural-decision note in CLAUDE.md (which stays â€” it describes permanent behaviour).

**Verified (2026-06-25 â€” first live exercise after 6 clean-but-unexercised runs):** the 6/25 `--no-dry-run` daily run printed `Pending entry orders (skipping new signals): ['GEV', 'GS']` in Phase 4 â€” proving the `get_open_orders()` fetch + not-held filter resolve against real book state (GEV/GS each had a working unfilled GTC entry and were not held) â€” **AND GS regenerated a BUY (score 0.698) that same run and was skipped**, yielding `Skipped pending orders: 1` in the Phase 5 summary. Both halves of the resolve criterion (the print proving the fetch path, and a real skip proving the dedup) fired on one run; no second bracket was stacked. The 6/15â€“6/24 runs all reported `Skipped pending orders: 0` with empty pending-entry sets (the motivating HPE/LRCX entries had been cancelled manually, and subsequent runs had only held-name or fresh-symbol BUYs), so this was the first run where a working-but-unheld entry coincided with a regenerated signal. See `docs/reviews/followups.md` (2026-06-15 item, resolved 2026-06-25).

### Config tuning â€” circuit-breaker limits 3%/7% â†’ 5%/10%

*(not a bug â€” a deliberate risk-appetite change, recorded here because the CLAUDE.md config table points to this entry for the rationale)* (`config/settings.py:RiskConfig`, `config/settings.yaml`): raised `circuit_breaker_daily_loss_pct` 0.03 â†’ **0.05** and `circuit_breaker_weekly_loss_pct` 0.07 â†’ **0.10**. **Rationale:** the system has two protection layers â€” per-position **ATR stops** (first line, individual position risk) and the **portfolio circuit breaker** (last line, correlated/systemic tail moves). 3% of a ~60%-semis/AI book whose names swing 3â€“5% intraday routinely is *ordinary* volatility, so the CB was doing first-line work the ATR stops already do â€” over-firing on noise and dumping the whole book (winners included) on a recoverable dip. A last-line tail brake should sit further out; 5% reserves it for "something is actually wrong." The 6/10 episode is the supporting evidence: it was a **dip, not a crash** â€” the noon flatten was right for that day (holding to the 6/10 close would have cost another âˆ’$9,278) but the book **bounced +$13,533 above the flatten price by the 6/11 close**, so the 3% trip + 24h halt sold the local low and sat flat through a V-recovery. Under 5% the book (â‰ˆâˆ’4.7% at the 6/10 close) would not have tripped and would have ridden the bounce. **The weekly was raised in tandem** so the pair stays coherent â€” at 5%/7% a single max-daily-loss day leaves only 2% of weekly headroom, making the weekly the de-facto daily brake on day 2; 5%/10% keeps them independent. **Caveat (do not over-update on the +$13.5k bounce):** that recovery is one observation â€” had 6/11 fallen another 4%, the 3% flatten would have looked protective and 5% reckless. The decision rides on the *principle* (CB = tail brake behind the ATR stops), not the n=1 outcome. The residual bluntness â€” even at 5%, a trip still flattens winners alongside losers via `flatten_all_longs` â€” is a separate, deferred code-level refinement ("trim to bring exposure under the limit" vs. "flatten everything"), tracked in `docs/reviews/followups.md` (2026-06-11). Tests: `tests/test_risk.py::TestCircuitBreaker` trigger tests now pin their own limit via monkeypatch (config-independent) so future tuning won't break them; full suite 447 passed.

## 2026-05-01

### Long-only close didn't cancel bracket children, leaving an orphaned STP that opened an unintended short

*(code complete 2026-05-01 â†’ verified retroactively 2026-06-04 via the SLV 2026-04-29 forensic reconstruction)* (`risk/order_manager.py:_close_long_position` / `_cancel_bracket_children`): before this fix, the long-only SELL path (`_close_long_position`) market-sold to flatten the position but did **not** cancel the entry bracket's still-live GTC children (the STP stop-loss and LMT take-profit legs). Those legs survived at IBKR against a now-zero-share position; a subsequent price move that triggered the orphaned STP would SELL into flat and **open an unintended short** â€” directly violating `allow_short_selling=False`. This is the *root* orphan-cancellation gap; the later 2026-05-21 entry ("Phase 4 long-only close doesn't cancel TRAIL orders") is a refinement of the *same* fix that extended the cancel filter to TRAIL legs.

**Canonical example â€” SLV, reconstructed 2026-06-04 from `fill_log` + `order_decisions` + `logs/daily/daily_run_20260427.log`:** (1) **2026-04-22** BUY 566 bracket entry, APPROVED, STP stop @ ~$65.08. (2) **2026-04-27** SELL signal flip â†’ `CLOSED_LONG`; the daily log shows the close went straight to `Placing MARKET order: SELL SLV x566` â†’ `Market order result: [Filled]` â†’ `CLOSED_LONG` with **no `Cancel request sent` line at all** â€” because `_cancel_bracket_children` did not yet exist on that date (the close pre-dated this very commit by 4 days). The STP leg was left live. (3) **2026-04-29** SLV fell to $64.96, tripping the orphaned STP â€” `fill_log` records the fill as `order_type='STP'`, 566 shares SELL â€” which opened a 566-share short. (4) **2026-04-30** BUY 566 MKT covered the short. The resulting `trade_log` round trip (`id=2002`) was a genuine short that the long-only reconciliation aggregator then mislabeled with reversed entry/exit timestamps (see the separate 2026-06-04 data-correction note); the short itself was this bug.

**Status:** `_cancel_bracket_children` introduced in commit `06b5361` (2026-05-01, bundled in the "Committing several days of code updates" commit â€” which is why this fix had no dedicated CHANGELOG entry until now). `_close_long_position` calls it before submitting the market close, cancelling open SELL bracket legs of type `("LMT", "STP", "STP LMT")` for the symbol. Commit `73aceb5` (2026-05-21) widened that filter to `("LMT", "STP", "STP LMT", "TRAIL")` â€” see the 2026-05-21 entry for the TRAIL-specific test (`test_cancel_bracket_children_cancels_trail_orders`) and the ASTS production incident.

**Verified (retroactively, 2026-06-04):** no SLV-class orphan short has recurred since the fix. Every SLV long-close after 2026-05-01 cancels its bracket children, and the only orphan-short round trip in `trade_log` (`id=2002`) is the pre-fix 2026-04-29 event whose entry pre-dated the cancellation feature. The detection query `SELECT * FROM trade_log WHERE exit_ts < entry_ts` (which surfaced the SLV row) returns zero rows after the data correction. Forward regression protection lives in the 2026-05-21 `test_cancel_bracket_children_cancels_trail_orders` test, which pins the cancel-filter membership.

**Correction (2026-06-05) â€” the 2026-06-04 "Verified" claim above was premature; the exit-before-entry corruption *regenerated* on the 2026-06-05 daily run.** The `_close_long_position` cancellation fix (the actual subject of this entry) is sound and unaffected â€” no *new* orphan short has been created since 2026-05-01. But the 2026-06-04 work also did a one-off **data correction** to `id=2002` (re-labelling it as a proper short with `entry_ts < exit_ts`), and that correction did **not** fix the *reconciliation aggregation logic* that produced the corrupt row in the first place. Worse, by changing `id=2002`'s `exit_exec_id` to the 4/30 MKT-buy leg, it freed the 4/29 STP-sell leg to be re-claimed: on 2026-06-05 `execution/reconciliation.py:_aggregate` re-paired the **same two April orphan-short fills** in the opposite direction (a "long" with entry 4/30 â†’ exit 4/29) and wrote `trade_log` id=2022 â€” `exit_ts < entry_ts` again, double-counting the âˆ’$974.39 loss against id=2002. The `uq_trade_live_exit` partial unique index keys only on `exit_exec_id`, so the reverse pairing (different exit leg) dodged dedup. **The generating logic was fixed 2026-06-05** in `execution/reconciliation.py:_write_round_trip`: (a) a hard `exit_ts <= entry_ts` guard rejects + WARNs any inverted pairing (`ReconcileResult.n_skipped_inverted`), and (b) a both-legs dedup via the new `data.database.live_trade_uses_exec_ids` â€” a fill already consumed as **either** an entry or exit leg of a live row can no longer be re-paired. `id=2022` was deleted; `WHERE exit_ts < entry_ts` returns zero rows again. Regression protection: `tests/test_reconciliation.py::TestInvertedRoundTripGuard` (2 tests) pins both the no-write and the stable-on-rerun behaviour. **Lesson: a data correction is not a fix** â€” this orphan-short reconciliation work stays out of "verified" until the *logic* fix has survived several daily runs without regenerating the row (tracked in `docs/reviews/followups.md`, 2026-06-05).
